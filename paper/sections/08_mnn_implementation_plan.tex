\section{MNN 実装計画}

最終実装は MNN CPU inference を使う。offline training は PyTorch、TensorFlow、
ONNX などを使ってよいが、論文で実装可能性を主張するには以下が必要である。

\begin{enumerate}
  \item 入力 tensor shape、10 秒統計スキーマ、系列長 $N$ を固定する。
  \item benign training split だけから normalization constant を凍結する。
  \item 候補モデルを ONNX 等へ export し、MNN converter で固定 shape に変換する。
  \item FP32 と、可能なら Int8/FP16 の MNN model を生成する。
  \item 同一入力で offline framework と MNN の anomaly score を比較し、許容誤差を定義する。
  \item model weight 表現、quantization metadata、入力統計列、normalization、
        threshold、score history を含む volume あたりの detector-data size を測定する。
  \item activation tensor、operator workspace、output tensor、inference slot は
        transient scratch として別に測定する。
  \item volume 数 $V$ を変え、共有 runtime/library、共有可能な weight、volume ごとに
        複製される state を分けて aggregate memory と 10 秒あたり CPU 時間を測定する。
  \item volume あたり detector-data が 500 KB を超える場合、feature 数、系列長 $N$、
        hidden size、architecture を縮小して再評価する。
\end{enumerate}

実装時の model interface は全候補で共通にする。PyTorch 側の \texttt{forward} は
正規化済み 10 秒統計列
$X \in \mathbb{R}^{B_{\mathrm{batch}} \times N \times D}$ を受け取り、
同じ shape の $\hat{X}$ だけを返す。anomaly score、しきい値判定、
channel-wise error 集計は MNN graph の外で実装する。feature slice は固定順序とし、
count/bytes、write ratio、mean LBA、mean length、窓間差分、
optional telemetry slot、padding を同じ index に保つ。ONNX/MNN export は
batch size 1、固定 $N$、固定 $D$ とし、動的軸を使わない。metadata-only 実験では
optional slot を zero-fill し、対応する loss weight と score weight を 0 にする。

この手順では、MNN ランタイムのコードサイズ、共有 library、共通 heap は
volume あたり 500 KB detector-data 予算から除外する。これは、約 2000 volume の展開では共有部分が
volume あたりに償却されるためである。一方で、モデルファイルが flash 上に常駐し
RAM に展開されない構成か、ロード時に RAM コピーされる構成かは target device に依存するため、
実装論文では両者を明記する。また、多数 volume で同一 model weight を共有できる場合でも、
各 volume の入力列、正規化状態、calibration threshold、score history は独立に必要になる
可能性が高い。したがって ``軽量モデルが 1 個動く'' ことをもって組み込み可能性とはせず、
想定 volume 数での総メモリと CPU スケジュールを device-fit claim の条件に含める。
報告では式 \eqref{eq:aggregate-cost} の $M_{\mathrm{shared}}$ と
$M_{\mathrm{w,shared}}$、$M_{\mathrm{w,volume}}$、$M_{\mathrm{state}}$、
$M_{\mathrm{scratch}}$ を分け、weight 共有ができる実装とできない実装の両方を
感度分析する。operator workspace が inference slot の再利用で共有できる場合も、
同時推論数や 10 秒 frame の締切で必要な slot 数 $Q$ が変わるため、単なる weight size ではなく
スケジューリング後の $M_{\mathrm{peak}}(V,Q)$ を測定値として採用する。
